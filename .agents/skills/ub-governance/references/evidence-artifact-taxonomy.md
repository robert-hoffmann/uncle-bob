# Evidence Artifact Taxonomy

Artifacts are selected by:

1. `evidenceLevel` (`L0` to `L3`)
2. `changeType` (`ui-feature`, `api-feature`, `bugfix`, `refactor`, `security`, `perf`, `a11y`, `migration`, `release`)
3. high-risk path impact from `high-risk-paths.yaml`

## 1) Artifact Buckets

1. correctness evidence
2. safety/static evidence
3. user-impact evidence
4. operability/release evidence
5. decision-memory evidence
6. claim-confidence evidence

## 2) Baseline by Evidence Level

| Level | Required Blocking Artifacts |
| ----- | --------------------------- |
| `L0` | command log, static/lint signal, build/syntax check (if applicable), validation record |
| `L1` | `L0` + unit/component outputs, coverage outputs, targeted smoke evidence when touched |
| `L2` | `L1` + expanded integration/contract/e2e evidence, security evidence, rollback plan, decision object |
| `L3` | `L2` + release checklist, rollout plan, monitoring references, post-deploy verification evidence |

## 3) Change-Type Modifiers

| Change Type | Additional Must-Meet Evidence |
| --- | --- |
| `ui-feature` | changed-view component/e2e coverage and accessibility outputs |
| `api-feature` | integration + contract validation + backward-compat notes |
| `bugfix` | regression-path evidence for reported defect path |
| `refactor` | unchanged behavior evidence and decision note when structural |
| `security` | security outputs, remediation note, abuse-path/permission evidence |
| `perf` | performance budget output and baseline comparison |
| `a11y` | accessibility outputs for changed pages/components |
| `migration` | migration rehearsal and rollback rehearsal evidence |
| `release` | release checklist, rollback runbook, monitoring evidence |

## 4) High-Risk Modifier (Immediate Blocking)

When touched paths match `high-risk-paths.yaml`, require:

1. ADR alignment evidence (`docs/adr/registry.json` linkage)
2. decision object fields: `impact`, `alignment`, `adrRefs`, optional `waiverRef`, `constraintsRef`
3. claim-register validation output when blocking rationale depends on claims

## 5) Validation Record Linkage

Every blocking run must produce a record matching `agent-validation-record.schema.json`.

Required linkage includes:

1. classification (`changeType`, `evidenceLevel`, `highRiskChange`)
2. environment metadata
3. produced artifact paths
4. decision fields for high-risk and higher-evidence runs

TDD fields in the validation record are consumed evidence inputs generated by testing workflows.
