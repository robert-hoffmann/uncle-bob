# Sprint 02 Evidence

## Normalized Lifecycle Summary

After Sprint 02, the workflow-facing references and guides now teach the same
high-level sequence:

1. research and discovery
2. self-contained PRD
3. roadmap generation and explicit approval
4. sprint-pack preparation so each `sprint.md` is execution-ready
5. sprint-folder materialization when needed
6. one active sprint at a time
7. final audit
8. retained note and explicit archive decision

The practical correction is that roadmap approval no longer implies immediate
execution readiness, and `init-sprints` no longer implies that placeholder
shells are acceptable sprint artifacts.

## User-Facing Resume Order

The normalized resume order is now:

1. `./roadmap.md`
2. the most relevant prior sprint `closeout.md`
3. the active or next sprint `sprint.md`
4. `./README.md`
5. `./prd.md` only when initiative-level context is still missing

## Guidance Versus Enforcement Notes

1. Sprint-pack preparation is now explicit in the contracts and user guide.
2. The helper guidance now states clearly that current helper commands do not,
   by themselves, guarantee execution-ready sprint PRDs.
3. Full helper and template enforcement of that lifecycle remains Sprint 03
   work.
